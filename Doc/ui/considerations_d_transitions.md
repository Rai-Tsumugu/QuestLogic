# D. UI遷移 (Transitions & Flow) に関する詳細考慮項目

ユーザーが目的を達成するまでの道のりを迷わせず、待ち時間のストレスを最小限にするためのフローとトランジションのチェックリストです。

## 1. 最短経路の設計
- [ ] 子供: アプリ起動（またはユーザー切り替え）から「最初の写真撮影（クエスト開始）」に至るまでのタップ数は最適化されているか？（理想は2タップ以内）
- [ ] 親: プッシュ通知をタップしてから「承認ボタン」を押すまでのフローはシームレスか？
- [ ] 子供画面から親画面への切り替え（PIN入力等）導線は、子供が誤って触らないようにしつつ、大人がすぐに呼び出せる導線になっているか？

## 2. ローディングと非同期処理の演出
- [ ] After画像を撮影後、**AIによる画像解析**の待ち時間（数秒〜十数秒）が発生する間、ユーザーを飽きさせない演出（例：魔法陣の処理中アニメーション、Tips表示など）はあるか？
- [ ] ネットワーク通信中のスケルトンローディング表示や、読み込み中を示すプログレスバーは画面のトーン＆マナーに合っているか？
- [ ] 裏側で画像のアップロードが行われている際、処理をブロックせずにUIの操作を継続できるか？（例：アップロード完了を待たずにホーム画面へ遷移させる等）

## 3. エラー時のリカバリ導線
- [ ] AI解析がエラーで失敗した際に、「もう一度やり直す」ボタンは明確に表示されるか？
- [ ] エラーメッセージは専門用語（例: "Network Exception"）ではなく、ユーザーの言語（例: 子「通信に失敗したみたいだ！少し待ってもう一度試してくれ！」、親「ネットワークに接続できませんでした」）で表現されているか？
- [ ] 物理デバイス（Smart-Gate Box）との通信（施錠・解錠）がタイムアウトした際、手動リトライやエラー報告の遷移先は確保されているか？

## 4. モーションとマイクロインタラクション
- [ ] 画面遷移（Push, Modal）の動きは一定のルールに基づいているか？（例: 親UIは下からのスライドイン、子UIは暗転・ドットワイプ等）
- [ ] 要素が切り替わる際の不自然なチラつき（フラッシュ）を回避するためのプレースホルダー設計はされているか？
